OAS + MBS: Why the MXFP4 Kernel is Unchanged

All OAS/MBS changes are pure software — the Tensor Core sees standard E2M1 + E8M0

Stage 0 · Quantisation preprocessing software
1
MBS — extract 8-bit mantissa of 6 / αmax128, get correction factor  c = 1 + mMBS8, pre-scale: X′ = X × c
2
OAS — modified E8M0 scale selection (reference 7 instead of 8) keeps αmax in (3.5, 7], reducing overflow from 50 % → 25 %
✦ Output is still E2M1 data + E8M0 scale — the standard MXFP4 format the kernel already expects
Stage 1 · MXFP4 GEMM (Tensor Core) ★ UNCHANGED ★
mma.m16n8k64  ·  FP4 × FP4 → FP32 accumulator
Sees the same E2M1 + E8M0 it always does — no modification needed
Stage 2 · Epilogue correction Vector Core
3
Undo the MBS pre-scale in the accumulator:
Cij ← Cij × σA,i × σB,j  where  σ = 1/c
Runs concurrently on Vector Cores; overhead < 1.56 % of Tensor Core peak
Core idea: OAS and MBS operate entirely before and after the kernel. The kernel input is always standard E2M1 + E8M0 — so no hardware modification is needed. “MBS is strictly a software optimization, requiring no hardware changes.” — paper
